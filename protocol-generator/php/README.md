## What it is

A native PHP peer with a **zero-runtime-Composer-dependency core**: the canonical-CBOR (ECF) codec,
Base58, the multicodec LEB128 varint, content-hash, peer-id, the §4.1 handshake / §6.5–§6.6 dispatch
machinery, capability authorization with chain attenuation + §5.7 delegation caveats, the §9.5 53-type
registry, the §9.5a CORE-TREE get/put/CAS/delete surface, the §10.1 core-register gate, the §4.10
resource-bounds checks, and the §7b concurrency gate are all hand-rolled in `src/`. Crypto comes from
**ext-sodium** + the stdlib `hash()` — both **bundled with PHP**, so `composer install` (no-dev) pulls
*nothing* onto the runtime path. The agility uint64 carrier comes from **ext-gmp**. No CBOR library
(none gives ECF canonicality — A-PHP-001), no FFI in the shipped floor peer.

- **Codec:** hand-rolled canonical CBOR (`src/Cbor.php`) over a binary PHP `string` (binary-safe —
  `strlen`/`substr`/`ord`/`unpack`; never routed through `mb_*`). Shortest-float minimization
  (f16 ⊂ f32 ⊂ f64, narrowest bit-exact round-trip, the four exact specials `F9 7E00`/`7C00`/`FC00`/
  `8000`; PHP `pack`/`unpack` has **no half-float code**, so f16 is hand-assembled from the binary64
  bits — A-PHP-004), **length-first-then-bytewise** map-key ordering on encoded key bytes (the trap
  stock CBOR libs fall into), recursive major-type-6 **tag rejection at any depth**, definite-length
  only. Base58 + LEB128 varint hand-rolled.
- **The uint64 carrier (the PHP headline — A-PHP-003):** PHP's `int` is **64-bit signed**
  (`PHP_INT_MAX = 2^63 − 1`) with silent overflow-to-float (lossy past 2^53), so the full CBOR
  integer head-form `[2^63, 2^64 − 1]` band cannot ride a native int. It is carried **uniformly
  through GMP** (ext-gmp) — one path, no int/GMP branching; the high band's 8 wire bytes are built
  from the GMP value (`gmp_export`), never an int cast. This is the OCaml int63 / C# `ulong` / TS
  `bigint` trap re-derived on a **3rd signed-int substrate** — PHP is in the *has-a-trap* camp, the
  opposite of Ruby's arbitrary-precision free pass. The `int.10/15/16/17` vectors are the proof.
- **Crypto:** Ed25519 sign/verify via **ext-sodium** (`sodium_crypto_sign_*`, libsodium —
  deterministic RFC-8032, so cross-impl signature byte-equality) + SHA-256/384 via the stdlib
  `hash()`. Both ship with PHP — **zero Composer/PECL dep, no FFI**. **Ed448 / SHA-384-agility is
  deferred** (A-PHP-002): libsodium has no Ed448 and PHP's stdlib exposes no other EdDSA source. The
  §9.1 floor (Ed25519 + SHA-256) is fully native and is the only path the corpus exercises; when
  agility lands the documented route is hybrid **ext-ffi** to the sibling C-ABI `ec_ed448_*` (the
  OCaml shape), opt-in so the floor peer stays FFI-free (SHA-384 itself is already native via `hash()`).
- **Identity / peer_id:** §1.5 canonical-form construction (Ed25519 → `hash_type=0x00`
  identity-multihash, digest *is* the raw 32-byte pubkey), per the §1.5 v7.65 table — **not** the
  stale §7.4 SHA-256 pseudocode. Lowercase `bin2hex` hex everywhere (dodges the A-CL-009 uppercase
  address-space-path trap). The seed-`0x11` peer_id `2KHoAk…` is **byte-identical to the cohort**.
- **Error model:** a `\Exception`-rooted `EntityCore\EntityCoreException` lattice (`CodecException`,
  …) — the canonical PHP fallible idiom (`throw`/`catch`), the same family as Ruby's exception tree,
  distinct from OCaml's `result` / C++'s `std::expected` / Elixir's tagged tuples. Protocol status is
  carried as a **value, never across an exception** (the cohort's status-as-value invariant); §5.2 /
  §5.2a request verification is a three-way verdict (ALLOW / unauthenticated→401 /
  authenticated-but-unauthorized→403 / unresolvable-identity→401).
- **Concurrency (the PHP seam — A-PHP-005/009):** a **single-thread non-blocking event loop** over
  `stream_select()` (what ReactPHP/Amp abstract over). Standard php-cli has **no userland threads**
  (ext-pthreads dead; ext-parallel is a ZTS-only non-core dep), so the dependency-free idiomatic
  primitive is the event loop — and it yields the **cleanest store-safety story in the cohort**: one
  handler runs to completion before the next is dispatched, so §4.8 is satisfied *structurally* (no
  data races, the §3.9 CAS needs **no lock**). §6.11 reentry is cooperative — a handler originating an
  outbound EXECUTE pumps the same loop until the reply correlates by `request_id` (no thread, no
  condvar). This is a genuinely **new point in the §7b concurrency taxonomy** — *single-threadedness as
  the structural-safety mechanism*, alongside actor-isolation / STM / manual locking. Honest caveat: a
  single thread is not parallel (CPU-bound signature bursts serialize), but the peer is IO-bound, so it
  is §7b-adequate. `TCP_NODELAY` is best-effort via ext-sockets when present (A-PHP-010).

## Install / consume (Composer)

The package coordinate is **`entity-core/protocol`** (A-PHP-006 — a Composer package is implicitly
PHP, so the redundant `-php` suffix is dropped; PSR-4 root namespace `EntityCore\` → `src/`). When
published to Packagist:

```sh
# the unpromoted pre-release needs an explicit stability flag (composer.json already sets
# minimum-stability=dev + prefer-stable; an @dev/@alpha constraint pins it for a consumer):
composer require entity-core/protocol:@dev
```

```php
<?php
require 'vendor/autoload.php';

use EntityCore\Cbor;
use EntityCore\PeerId;

$bytes = Cbor::encode($value);        // canonical ECF bytes (binary string)
$value = Cbor::decode($bytes);        // byte-cursor decode over the binary string
$pid   = PeerId::fromPublicKey(0x01, $pub);   // §1.5 canonical peer_id
```

The **full peer + transport** (`EntityCore\Peer`, `Transport`, the store + capability surface) is
driven via the **`bin/peer`** host script (`bin/peer --port N [--name NAME] [--debug-open-grants]
[--validate]`, emits `LISTENING …`). That peer-layer surface is the S4 conformance/oracle **driver** —
not part of a frozen public API at `0.1.0-pre`. PHP has no module-private keyword, so a hard
public-surface freeze (relocating the in-repo test clients that reference internals by name) is a
publish-prep pass deferred until the surface is frozen against a first external consumer — the honest
`-pre` state, mirroring the Ruby / OCaml `.mli` / Zig `root.zig` deferral. Internal units (`Varint`,
`Base58`, `Cbor` framing, the type-floor render table) may churn without a semver bump.

## Versioning

The cohort release line is `0.1.0-pre`. **Composer's version grammar does not accept the literal
`0.1.0-pre`** — `pre` is not one of Composer's recognized stability keywords (`alpha`/`beta`/`RC`/
`dev`/…), so `composer validate` rejects it with `Invalid version string "0.1.0-pre"` (verified
in-container, Composer 2.7 / PHP 8.3). This contradicts the S1 A-PHP-006 prediction that Composer
accepts the SemVer-dash natively — it is the PHP analogue of the RubyGems `0.1.0.pre.pre` (A-RUBY-010)
and CL ASDF (A-CL-010) version-grammar surprises (now A-PHP-012). The idiomatic Composer resolution is
to **omit the `version` field entirely** — Composer's own guidance is to let the registry infer the
version from the VCS git tag (the `0.1.0-pre`/`v0.1.0` tag at publish time), which `composer validate
--strict` reports **valid**. So `composer.json` carries no `version`; the `0.1.0-pre` label lives in
this README / CHANGELOG / `status/PHASE-S5.md`, and the publish-time git tag is the registry
coordinate. Promotes to `0.1.0` when (a) S4 is fully green [met] **and** (b) ≥1 external consumer
confirms the peer [not yet met].
