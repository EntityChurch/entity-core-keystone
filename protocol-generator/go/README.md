[`entity-core-keystone`](../../README.md) `/entity-rosetta` pipeline.

## What it is

A pure-Go peer (**zero third-party modules — stdlib-only, `go.sum` is empty**):
canonical-CBOR (ECF) codec, Ed25519 identity + deterministic signatures, capability
authorization with chain attenuation, the handshake / dispatch machinery, the §9.5 53-type
registry, and the CORE-TREE get/put/CAS/delete surface — no FFI, no cgo, no libsodium.
`go build` / `go test` pull nothing off the network.

- **Codec:** hand-rolled canonical CBOR (`internal/cbor`) — shortest-float minimization
  (f16 ⊂ f32 ⊂ f64, narrowest bit-exact round-trip), length-then-lex map-key sort on
  encoded key bytes, definite lengths, no-duplicate-keys, recursive major-type-6 tag
  rejection on decode (→ `400 non_canonical_ecf`, §6.3), full `uint`/`nint` 0..2⁶⁴−1
  range. LEB128 multicodec varints (`internal/varint`) + Base58 (`internal/base58`)
  hand-rolled (neither is in the stdlib). Go's stdlib ships no CBOR; the one credible
  third-party candidate (`fxamacker/cbor`) was considered and rejected — see A-GO-001.
- **Crypto:** Ed25519 via `crypto/ed25519` (stdlib, audited, RFC-8032 deterministic
  signing — no RNG to sign); SHA-256 via `crypto/sha256`, SHA-384 (agility hashing) via
  `crypto/sha512`. *Ed448 / crypto-agility higher bar is not yet supported — Go's stdlib
  has no Ed448 and `golang.org/x/crypto` has none either; no audited pure-Go Ed448 in a
  reviewed channel exists. The planned path is hybrid (native Ed25519 + FFI Ed448 via cgo).
  See [`status/SPEC-AMBIGUITY-LOG.md`](status/SPEC-AMBIGUITY-LOG.md) A-GO-002 (mirrors Zig
  A-ZIG-002 / OCaml A-OC-002).*
- **Integers:** CBOR head-form carrier over native `uint64`/`int64` — the full §3.2 range
  maps directly, no BigInt ceremony (contrast TS) and no 63-bit trap (contrast OCaml). The
  `[-2⁶⁴,-1]` `nint` band is carried with explicit `uint64` arithmetic (A-GO-003).
- **Concurrency:** goroutines + channels. One reader goroutine per connection demuxing
  `EXECUTE_RESPONSE` by `request_id`; each inbound EXECUTE dispatched on its own goroutine
  so it never blocks outbound (§6.11). The §4.8 content store is `sync.RWMutex`-guarded
  (read-mostly: resolves are reads, binds are writes); emit consumers fire **outside** the
  lock. `SetNoDelay(true)` on every connection (the §7b throughput floor). The store was
  built race-safe from S3 — the Zig/Common-Lisp store-race lesson, pre-resolved.

## Build / test / run-conformance (in-container, offline)

All build/test/conformance runs happen in the pinned `entity-core-keystone/go:latest`
Podman image (`fedora:43` + `golang-1.25.10` via dnf, a reviewed distro channel). The peer
is stdlib-only, so every run is fully sealed-offline (`--network=none`). There is no host Go
dependency. The module root is `protocol-generator/go/src/` (`go.mod` lives there).

```sh
# Build (stdlib-only — go.sum stays empty)
podman run --memory=4g --memory-swap=4g --pids-limit=2048 --cpus=4 --rm --network=none -v "$PWD":/work:Z -w /work/protocol-generator/go/src \
  entity-core-keystone/go:latest go build ./...

# Unit + integration tests: S2 69/69 codec vectors + the §9.5 53-type byte-diff
# (TestCoreTypeRegistryByteIdentical) + the S3 11/11 loopback smoke
podman run --memory=4g --memory-swap=4g --pids-limit=2048 --cpus=4 --rm --network=none -v "$PWD":/work:Z -w /work/protocol-generator/go/src \
  entity-core-keystone/go:latest go test ./...

# gofmt + go vet (the Go lint floor)
podman run --memory=4g --memory-swap=4g --pids-limit=2048 --cpus=4 --rm --network=none -v "$PWD":/work:Z -w /work/protocol-generator/go/src \
  entity-core-keystone/go:latest sh -c 'go vet ./... && gofmt -l .'

# S4 live-peer conformance (validate-peer --profile core). The Go oracle ELF and this
# peer run together inside the go:latest image, sharing one loopback, offline.
podman run --memory=4g --memory-swap=4g --pids-limit=2048 --cpus=4 --rm --network=none -v "$PWD/protocol-generator":/work/protocol-generator:Z \
  -v "$PWD/protocol-generator/go/output/s4-oracles":/work/output/s4-oracles:Z \
  -e GOWORK=off entity-core-keystone/go:latest sh /work/protocol-generator/go/run-s4.sh
```

The S4 oracle (`output/s4-oracles/validate-peer`, gitignored) is a `CGO_ENABLED=0` Go ELF
built from `entity-core-go` `75c532e` **in a temp dir OUTSIDE the oracle tree** (`git
archive`, never cd'd into it). Rebuild it per the runbook in
[`status/CONFORMANCE-REPORT.md`](status/CONFORMANCE-REPORT.md) §Reproduce when the oracle

## Use

The public API is the module root, `package entitycore` (import path
`github.com/entity-core/entity-core-protocol-go`). Codec internals live under
`internal/{cbor,base58,varint}` — compiler-enforced encapsulation, not importable by
consumers. Two tiers:

```go
import ec "github.com/entity-core/entity-core-protocol-go"

// Tier 1 — codec island: encode/verify ECF, content-hash, peer-id, sign/verify
bytes, err := ec.EncodeECF(entity)              // canonical ECF bytes (T, error) idiom
h, err := ec.ContentHash(bytes)                 // content-addressing hash
id, err := ec.NewIdentity(seed)                 // Ed25519 identity + canonical peer_id
sig := id.Sign(msg); ok := id.Verify(msg, sig)  // deterministic Ed25519

// Tier 2 — full peer (package github.com/entity-core/entity-core-protocol-go/peer)
p, err := peer.NewPeer(peer.Config{Seed: seed, OpenGrants: false})
err = peer.Serve(p, "127.0.0.1:7778")
```

Exact entry points are godoc-documented on each exported surface. Internal units
(`cbor`, `base58`, `varint`) are implementation detail under `internal/` and may churn
without a semver bump. The `peer` subpackage's exported surface is documented but not yet
frozen against a semver lock — deferred to publish-prep / first external consumer (see
[`status/PHASE-S5.md`](status/PHASE-S5.md) §Public-surface).

## Distribution (when published)

Go has **no central package registry** (no crates.io/npm/NuGet equivalent). "Publishing" is
a git tag at a reviewed commit; consumers `go get
github.com/entity-core/entity-core-protocol-go@vX.Y.Z` and the checksum is recorded in
*their* `go.sum` + the public `sum.golang.org` transparency log — decentralized and
checksum-pinned by design (a supply-chain-friendly property). The stdlib-only posture means
a consumer inherits **zero transitive deps** — the only pin they take on is the Go toolchain
version (`go 1.25`). See [`status/PHASE-S5.md`](status/PHASE-S5.md) §Operator-handoff and
[`status/ARCHITECTURE-REVIEW.md`](status/ARCHITECTURE-REVIEW.md) §Publishing-options for the
go.mod-path / git-tag nuance and the in-repo-vs-standalone-repo decision surface
(operator-decides; not acted on by the pipeline).
