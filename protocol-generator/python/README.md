# entity-core-protocol-python

A **clean-room Python core-protocol peer** for the Entity Core protocol (V7), generated by
the [`entity-core-keystone`](../../README.md) `/entity-rosetta` pipeline.

> **Clean-room.** This is an INDEPENDENT reimplementation derived purely from the V7
> specification — **no file under any `entity-core-py` checkout (the hand-written Python
> sibling) was opened, read, or referenced while authoring it.** The value is an
> independent **adoption + generator-independence cross-check**: a from-scratch Python peer
> that byte-agrees with the Go conformance oracle. Because the result is same-as-sibling,
> the spec-*refinement* signal is bounded — the contribution is idiom breadth +
> convergence evidence, not novel spec discovery (the well is dry, as expected). See
> [`arch/PROFILE-RATIONALE.md`](arch/PROFILE-RATIONALE.md) and the status reports.

## What it is

A Python peer with **exactly one runtime dependency** — `cryptography==48.0.0` for the
EdDSA + SHA-2 crypto path; everything else is hand-rolled in-repo (canonical-CBOR / Base58 /
varint) or stdlib (`hashlib` / `threading` / `socket` / `struct`):

- **Codec:** hand-rolled canonical CBOR / Entity Canonical Form (`_cbor.py`) — shortest-float
  minimization (f16 ⊂ f32 ⊂ f64), length-then-lexicographic map-key sort on the *encoded*
  key bytes, recursive major-type-6 tag rejection on decode, full 0..2⁶⁴−1 head-form integer
  range. CPython's arbitrary-precision `int` carries the full u64 range with no native-int
  trap. Wire bytes are `bytes`, never `str`.
- **Crypto — native-full-agility, incl. Ed448, no FFI.** Both **Ed25519 AND Ed448**
  sign/verify (and SHA-256/384) come natively from `cryptography`'s bundled OpenSSL 3.5.x
  through one generic raw-key EdDSA surface — placing Python in the **native-full-agility
  class** (with Haskell / Ruby), NOT the hybrid-FFI class. No PyNaCl, no libsodium, no
  second crypto source, no C-ABI in the manifest. (The container build hard-asserts both
  curves at image-build time so a trimmed OpenSSL fails loudly.)
- **Identity:** §1.5 canonical-form `peer_id` (Base58 of `varint(key_type) ||
  varint(hash_type) || digest`), `content_hash` construction, deterministic RFC-8032
  signatures → cross-impl signature byte-equality.
- **Peer machinery (`entity_core.peer`):** §4.1 handshake, §6.5/§6.6 dispatch, capability
  authorization with chain attenuation + §5.7 caveats, the §9.5 53-type registry, the
  in-memory address-space store with §3.9 CAS, CORE-TREE get/put/CAS/delete, and the §6.13
  register / §6.9a owner-cap / §7a conformance surface. **Thread-per-connection** concurrency
  (`threading`, not asyncio) with a `Lock`-guarded store and a `Condition`-based §6.11
  reentrant request-id demux.

The extension surface (TREE, CONTENT, IDENTITY, ATTESTATION, QUORUM, …) stops at the
dispatcher interface — this is a **core** peer (V7 Layers 0–4). Community handlers install
above that boundary.

## Install

```bash
pip install entity-core-protocol-python      # (after first publish; deferred — see Status)
```

Then:

```python
import entity_core

# codec
wire = entity_core.encode({"hello": b"world"})
obj  = entity_core.decode(wire)

# identity + signatures (Ed25519 / Ed448 native)
pid  = entity_core.format_peer_id(...)         # §1.5 canonical peer_id
sig  = entity_core.sign_entity(seed, ecf_bytes)

# the full peer
from entity_core.peer import Peer, listen
```

**Public surface (Tier 1):** the codec / identity / signatures / error lattice re-exported
from `entity_core.__all__`. **Tier 2:** the full peer under `entity_core.peer`. Leading-
underscore modules (`_cbor` / `_base58` / `_varint`) are internal and may churn.

## Build & run in-container (host podman, offline)

The peer builds and the codec gate runs in the `python-toolchain` image
(`containers/python-toolchain/Containerfile`, fedora:43 + CPython 3.14.5 + the one
HASH-pinned `cryptography` wheel). The offline core image carries **no** test framework, so
the codec gate runs under a stdlib zero-dep runner (A-PY-010):

```bash
# Codec conformance — 69/69, sealed-offline:
podman run --memory=4g --memory-swap=4g --pids-limit=2048 --cpus=4 --rm --network=none -v "$PWD":/work:Z \
  -w /work/protocol-generator/python entity-core-keystone/python-toolchain:latest \
  sh -c 'PYTHONPATH=src python tests/conformance/harness.py'

# Full --profile core gate (needs the vendored Go oracle binaries at output/s4-oracles/;
# see status/PHASE-S4.md "Oracle-build isolation"):
cd protocol-generator/python && ./run-s4.sh

# origination-core 3/3 (reference-peer-gated):
cd protocol-generator/python && ./run-origination-core.sh
```

### The peer host (`entity-core-peer` / `python -m entity_core.host`)

The standalone host is the conformance / oracle driver `validate-peer` dials. The installed
wheel exposes it as the console-script **`entity-core-peer`**; offline (no wheel) it runs as
`python -m entity_core.host`. Flags:

| Flag | Meaning |
|---|---|
| `--name NAME` | Load the Ed25519 identity from `~/.entity/peers/NAME/keypair` (entity-core PEM = base64 of a 32-byte seed) — gives a **stable cross-run peer_id**, which is what makes the multisig ACCEPT path (`valid_2of3_peer_signed_accepted`) exercisable rather than env-skipped. |
| `--port N` | TCP port to listen on (`0` = auto-assign). Prints `LISTENING <port>` so a harness learns the bound port. |
| `--seed HEX` | A 64-hex (32-byte) Ed25519 seed (alternative to `--name`). |
| `--validate` | Bootstrap the §7a `system/validate/*` conformance handlers (the `validate-peer --validate` path). |

```bash
entity-core-peer --name conformance --port 7778 --validate
```

## Build the wheel + sdist (operator / publish-prep)

The offline core image ships only the runtime dep — no PEP 517 build backend. Build in a
thin network-on layer (per `status/PHASE-S5.md`), e.g.:

```bash
podman run --memory=4g --memory-swap=4g --pids-limit=2048 --cpus=4 --rm -v "$PWD/protocol-generator/python":/pkg:Z -w /pkg \
  entity-core-keystone/python-toolchain:latest \
  sh -c 'pip install build && python -m build'
# → dist/entity_core_protocol_python-0.1.0-py3-none-any.whl + .tar.gz
```

Publishing (`twine upload`) is an **operator decision after review** — deferred per the
keystone lifecycle (no auto-publish, no auto-tag). See `status/PHASE-S5.md` §operator-handoff.
