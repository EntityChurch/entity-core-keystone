[`entity-core-keystone`](../../README.md) `/entity-rosetta` pipeline. The cohort's **first

backtracking, and the cut — and the most paradigm-distant probe attempted: a relational
program model, not an imperative or functional one. The codec/crypto **byte-floor is sourced
over the C-ABI** (`codec_strategy = "ffi"`); **Prolog owns the relational core** (the §5

## What it is — the convergent logic layer

Every prior peer (C#, TS, OCaml, Elixir, Zig, Common Lisp, Swift, Haskell, Java, Ruby, C, Ada)
is, at bottom, an *imperative or functional* program: it computes a value or executes a
sequence. Prolog is neither — its computation model is **search over a relational model**. The
revival question this peer answers (`status/HANDOFF-PROLOG-REVIVAL.md`) was **not** "does the
byte-floor express as logic?" (it doesn't, and shouldn't — that's the FFI floor's job) but
"**does the protocol's operational semantics express as a convergent logic layer?**" — and the
answer is an emphatic yes. Concretely:

- **§5.5 capability-chain verification is a recursive relation** (`ec_capability.pl`):
  `verify_chain/3` over parent pointers, where **conjunction-failure IS the deny** — no boolean
  `good` flag threaded through a loop, no `return DENY`. `\+ verify_capability_chain(...)` reads
  the relation's failure directly as `authz_deny`.
- **§5.2 auth/authz trichotomy is guarded clause heads** (`verify_request/4`): each verdict
  (`authn_fail` / `authz_deny` / `chain_too_deep` / `allow`) is a distinct clause; the first
  whose body holds wins. No nested if/else verdict ladder.
- **§6.5/§6.6 dispatch is a multi-head clause table** (`ec_peer.pl` `handle_op/4`): each
  `(handler, operation)` pair is its own clause head, "unknown → 501" is the final catch-all
  clause. **The clause database IS the router** — adding an operation is adding a clause.
- **§3.9 store is the clause database itself** (`ec_store.pl`): the content store and entity
  tree are dynamic predicates (`content_fact/3`, `tree_fact/3`); the store operations ARE
  `assertz`/`retract`.

The honest findings — including the one path where Prolog's *failure* model is too weak (the
§5.5 401 carve-out, A-PL-006), and where the peer reads as "C with `:-`" (the FFI floor) — are
written up in **[`status/IDIOM-FINDINGS-SYNTHESIS.md`](status/IDIOM-FINDINGS-SYNTHESIS.md)**.
That synthesis is the actual deliverable of this peer.

## The FFI-floor architecture (Prolog logic core + C-ABI byte-floor)

SWI `library(crypto)` 9.2.9 exports **no** `ed25519_*` / `ed448_*` predicates at all
(A-PL-002/003, resolved NEGATIVE — SHA-256/384 work via `crypto_data_hash/3`, signing does not),
and the canonical-CBOR shortest-float ladder is irreducibly imperative bit arithmetic
(A-PL-004). Both are legitimately the C-ABI's job. So the peer splits cleanly:

```
  ┌─────────────────────────────────────────────────────────────┐
  │  Prolog relational core  (prolog/ec_*.pl)                     │
  │    ec_capability  — §5 chain walk (recursive relation)        │
  │    ec_peer        — §6.5 dispatch (multi-head clause table)   │
  │    ec_store       — §3.9 store (the clause database)          │
  │    ec_transport   — L4 TCP + §6.11 request_id demux           │
  │    ec_cbor        — §4.1 data-VALUE canonical CBOR (A-PL-013) │
  └───────────────┬─────────────────────────────────────────────┘
                  │  ec_codec.pl  — deterministic (once/1) predicate API
                  ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  C-ABI byte-floor  (c/ec_codec_pl.c → ec_codec_pl.so)         │
  │    use_foreign_library/1 shim over libentitycore_codec (1.1): │
  │    canonical-CBOR entity framing, content_hash, peer_id,      │
  │    Ed25519/Ed448 sign/verify/seed→pubkey, SHA-256/384         │
  └─────────────────────────────────────────────────────────────┘
```

No external `ffi` pack — the shim is a plain `use_foreign_library/1` load. The C-ABI floor
treats the entity `data` value as opaque pre-encoded bytes (A-PL-013), so the peer **still owns
the data-value canonical CBOR** (`ec_cbor.pl`); the 53/53 type-registry byte-diff proves that
Prolog data-value canon matches the cross-impl encoder.

## Install / use

The peer is an **SWI pack** (`pack.pl`), the Prolog analogue of CL's `.asd` / a gemspec /
`Cargo.toml`. The module set under `prolog/` is the public surface (SWI has no module-private
keyword beyond each module's export list, so the export lists ARE the contract):

```prolog
% Make the pack discoverable, then use the modules.
:- attach_packs('/path/to/protocol-generator/prolog', [search(first)]).
% (or, once published to a pack registry: ?- pack_install(entity_core_protocol).)

% ── Tier 1 — codec island: deterministic ECF + crypto over the C-ABI floor ──
:- use_module(ec_codec).
?- ec_codec:ec_content_hash("test/v1", DataBytes, Hash33).   % canonical content_hash
?- ec_codec:ec_ed25519_sign(Seed32, MsgBytes, Sig64).        % deterministic Ed25519

% ── Tier 2 — full peer ──
:- use_module(ec_peer), use_module(ec_transport).
?- make_peer([seed(Seed), open_grants(false)], Peer),
   start_listener(ec_peer:serve_goal(Peer), 7777, _Sock-_Port).
```

The conformance host driver is `prolog/ec_host.pl` (`--port`, `--debug-open-grants`,
`--validate`; emits `LISTENING …`). It is the S4 conformance driver — **test/conformance
tooling, not a stable consumer surface.**

## Distribution (when published)

SWI's community channel is the **pack registry** (`pack_install/1` resolves a pack by name from
the registry, or directly from a git URL / tarball). "Publishing" registers the pack or points a
`download(URL)` at the release archive at the reviewed tag. There is no upload-an-artifact step
distinct from registering the source location — closer to CL's Quicklisp (git-indexed) than to
RubyGems (a real upload registry). The peer is consumed today via direct `attach_packs/2` from
the worktree, fully offline.

**Version-string gotcha (A-PL-019):** SWI's pack version grammar (`prolog_pack:is_version/1`)
is dotted-**numeric** only — stricter than SemVer *and* stricter than RubyGems. It rejects
`0.1.0-pre` / `0.1.0pre` / `0.1.0-alpha.1` (all INVALID; only `0.1.0` is VALID, verified
in-container). So `pack.pl` carries the parseable `0.1.0` while the `0.1.0-pre` pre-release LINE
lives in this README + [`CHANGELOG.md`](CHANGELOG.md). This is the SWI analogue of Common Lisp's
ASDF A-CL-010 and Ruby's RubyGems A-RUBY-010 — the third cohort ecosystem whose version grammar
disagrees with the SemVer dash, and the strictest (no pre-release channel at all).
