# type_system core-vs-extension classification — v7.71 oracle, 150 demanded types
#
# Source of demand: entity-core-go types.RegisterCoreTypes + connect/tree handler
#   types == validate-peer runTypeSystem registry == compare-types generateLocalTypes.
#   Rendered byte-exact to test-vectors/v0.8.0/type-registry-vectors-v1.{cbor,diag}
#   (150 entries) by protocol-generator/shared/tools/dump-type-registry.
#
# Ruling basis (user): a CORE protocol peer publishes core + operational
#   + the type-system BOOTSTRAP only. Extensions bring their own types+handlers+
#   functionality when installed; the registry grows as extensions are added. The
#   *type extension* (validation/merging) is NOT a core requirement. Publishing a
#   type *shape* != implementing the extension. See memory: type-registry-core-vs-extension,
#   type-registry-render-design.
#
# G4 (refined, escalate to arch): RegisterCoreTypes conflates all extension vocab
#   AND the type-extension into its "ratified-core" floor. The 97 extension types
#   below over-fail a correct core peer; exclude from the core verdict (same as the
#   handlers-extension exclusions). Only system/substitute/* is already provisional
#   (WARN) oracle-side; the other 93 are hard FAILs that must be excluded.
