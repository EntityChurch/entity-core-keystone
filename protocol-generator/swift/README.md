## What it is

A pure-Swift peer: canonical-CBOR (ECF) codec, Ed25519 identity + deterministic signatures,
capability authorization with chain attenuation, the §4.1 handshake / §6.5 dispatch machinery,
the §9.5 53-type registry, the §9.5a CORE-TREE get/put/CAS/delete surface, and the v7.74 Phase B
foundations (register / outbound closure / emit / peer-owner cap). **One pinned SwiftPM
dependency** — Apple's audited [swift-crypto](https://github.com/apple/swift-crypto) for
Ed25519 + SHA-2; CBOR / Base58 / varint are hand-rolled in-repo. No FFI on the core path.

- **Codec — `native`, hand-rolled canonical ECF (zero CBOR dependency):** shortest-float
  minimization (f16 ⊂ f32 ⊂ f64, narrowest bit-exact round-trip), length-then-lexicographic
  map-key sort **over encoded UTF-8 key bytes**, recursive major-type-6 tag rejection on decode
  (tags are not part of ECF), full `uint64`/`nint` 0..2⁶⁴−1 range, raw-byte fidelity. No Swift
  CBOR library gives ECF's guarantees (none distinguishes float `1.0` from int `1` by type, none
  does length-then-lex key ordering, none recursively rejects tags) — the same A-005 finding
  every prior native peer hit, re-confirmed for Swift. LEB128 varints + Base58 (Bitcoin
  alphabet) hand-rolled (neither is in the stdlib or swift-crypto).
- **Crypto:** Ed25519 via `Curve25519.Signing` (swift-crypto, RFC-8032 deterministic — no RNG
  for signing); SHA-256/384/512 via swift-crypto. Cross-impl signature byte-equality.
- **Integers:** the CBOR head-form carrier holds the full 0..2⁶⁴−1 range over native `UInt64` —
  the integer-width trap (no `Int64` clamp; proven by the `2^64-1` / `2^63` selftests).

## Swift-specific notes (why peer #7 exists)

Swift was chosen to probe two **unsaturated discovery axes** that the prior six peers
(C#/TS/OCaml/Zig/Elixir/Common-Lisp) didn't cover:

- **String / UTF-8-byte discipline (A-SW-002, the headline).** Swift `String.count` counts
  *grapheme clusters*, not bytes, and `String` is **not** random-access by `Int`. Every wire
  operation therefore runs on the UTF-8 byte view (`String.utf8` → `[UInt8]`), never on
  `String`: the CBOR text-string length is `String.utf8.count` (`"café"` → head `0x65`, 5 bytes,
  not `0x64`/4 graphemes); the map-key lexicographic sort runs over the **encoded UTF-8 key
  bytes** (`[UInt8]` comparison), never Swift's Unicode-canonical/locale-sensitive `String`
  ordering. Using `String.count` for a length or `String` ordering for a key would silently
  diverge from the wire for any non-ASCII data. This is the codec-discipline judgment call that
  defines the Swift port; resolved/informational (the spec is unambiguous — bytes/UTF-8).
- **ARC + structured concurrency.** Codec data are value types (`struct`/`enum`, copy semantics,
  `Sendable`, cycle-free by construction); the live peer uses an **`actor`-based store** so the
  §7b store-race is a *compile error*, not a runtime crash. The §7b finding that this peer
  surfaced (distinct from the cohort's): Swift's **structured-concurrency bounded cooperative
  thread pool is hostile to blocking syscalls** — a blocking `read()`/`accept()` pins a pool
  thread for its whole idle wait, so under connection churn the `Task.detached { blockingRead() }`
  shape exhausts the pool and new accepts time out. Fix: run each blocking I/O call on a
  dedicated `Thread` (the raw-socket analogue of thread-per-connection), keeping the cooperative
  pool free. Reusable generator guidance for any actor/async target doing raw blocking sockets.

## Install / use

Swift has **no central binary package registry** — SwiftPM resolves dependencies from a git URL
+ version tag (Swift Package Index is a discovery index over git, not a binary host).
"Publishing" is a reviewed semver git tag that consumers pin in their own `Package.swift` /
`Package.resolved`.

Add the package dependency (when published, by repo URL + version):

```swift
// Package.swift (consumer)
dependencies: [
    .package(url: "https://github.com/<org>/entity-core-protocol-swift.git", from: "0.1.0"),
],
targets: [
    .target(name: "MyApp", dependencies: [
        .product(name: "EntityCoreProtocol", package: "entity-core-protocol-swift"),
    ]),
]
```

```swift
import EntityCoreProtocol

// Tier 1 — codec island (no peer): encode / hash / identity
let entity = try Model.make(type: "system/peer", data: .map([("name", .text("alice"))]))
let bytes  = try CBOR.encode(entity.value)                       // canonical ECF bytes
let hash   = try ContentHash.contentHash(type: "system/peer",
                                         data: .map([("name", .text("alice"))]))
let id     = try Identity(seed: seed)                            // Ed25519 identity + peer_id

// Tier 2 — full peer (boot a listener; the builder API)
let peer   = try await Peer(seed: seed,
                            seedPolicy: .standard(),             // §6.9a identity→capability seed policy
                            conformanceHandlers: false)          // §7a system/validate/* opt-in, OFF by default
let server = try await Server(peer: peer, port: 7777)
await server.start()
```

The `Peer` builder is the single public boot surface: `seedPolicy:` selects the §6.9a
identity→capability map (`.standard()`, `.debugOpen()` = the deprecated degenerate `default → *`,
or `.of([...])` from a [seed-policy JSON](../shared/seed-policy/) file), and
`conformanceHandlers:` opts the peer into the §7a `system/validate/{echo,dispatch-outbound}`
handlers (off by default — `dispatch-outbound` is a standing dialer, never live in production).

## Crypto availability note

swift-crypto is the source on **Linux** (CryptoKit is an Apple-platforms-only framework that
does not exist on Linux — A-SW-003). swift-crypto implements the *exact CryptoKit API* and is
audited + BoringSSL-backed, so the same source compiles against CryptoKit on Apple platforms.
**Ed448 (key_type `0x02`, the crypto-agility *higher bar*) is deferred** (A-SW-001) — swift-crypto
/ BoringSSL omits Ed448 and no audited pure-Swift Ed448 exists; the planned shape is hybrid
(native Ed25519 + FFI Ed448 via `libentitycore_codec`, exactly the resolved OCaml A-OC-002
pattern). This does **not** affect the Ed25519 / SHA-256 §9.1 conformance floor, which is fully
native and complete.
