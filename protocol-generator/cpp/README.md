# entity-core-protocol-cpp

A native-**C++23** *core protocol peer* for the Entity Core protocol (V7), generated by the
[`entity-core-keystone`](../../README.md) `/entity-rosetta` pipeline. A **release "reach" peer**
(systems / games / embedded ecosystem coverage), derived **fresh in S1** from the V7 spec
(`spec-data/v7.75`) in the cohort's **RAII / `std::expected` / template** idiom — smart-pointer
ownership with deterministic destructors (no manual `new`/`delete`/`free`), a value-based
`std::expected<T, E>` error channel (exceptions reserved for programmer-error only), and
generic-programming where idiomatic. Its **closest analog is the C peer** (`protocol-generator/c/`)
— same libsodium crypto, same hand-rolled canonical ECF codec, same systems-language memory concerns
— but a **deliberately distinct idiom point**: where the C peer owns every allocation by hand through
raw `malloc`/`free` and a return-code/out-param error model, the C++ peer exercises RAII ownership,
`std::expected` results, and move semantics — the shapes a C++ developer actually writes.

## What it is

A native C++ peer whose **only third-party runtime dependency is libsodium** (Ed25519 + SHA-256):
the canonical-CBOR (ECF) codec, Base58, the multicodec LEB128 varint, content-hash, peer-id, the
§4.1 handshake / §6.5–§6.6 dispatch machinery, capability authorization with chain attenuation +
§5.7 delegation caveats, the §9.5 53-type registry, the §9.5a CORE-TREE get/put/CAS/delete surface,
the §10.1 core-register gate, the §4.10 resource-bounds checks, and the §7b concurrency gate are all
hand-rolled in `src/`. No CBOR library (none gives ECF canonicality — A-CPP-001), no FFI to the
sibling C-ABI codec for the codec itself (that codec is kept only as a free byte-for-byte cross-check
oracle — `test/ffi_xcheck.cpp`, A-CPP-001).

- **Codec:** hand-rolled canonical CBOR (`src/ecf.cpp`/`.hpp`) — shortest-float minimization
  (f16 ⊂ f32 ⊂ f64, narrowest bit-exact round-trip, the four exact specials `F9 7E00`/`7C00`/`FC00`/
  `8000`), RFC-7049 **length-first-then-bytewise (CTAP2)** map-key ordering (`std::sort` over encoded
  key bytes — differs from RFC-8949 §4.2 bytewise, the trap stock CBOR libs fall into), recursive
  major-type-6 **tag rejection at any depth** → `400 non_canonical_ecf`, definite-length only.
  **Full uint64 / −2⁶⁴ head-form via native `std::uint64_t`** — the cleanest int story in the cohort
  alongside C / Zig / Rust (no `ulong`/int63 special-casing). The decoder returns
  `std::span<const std::byte>` / `string_view` views into the input buffer (zero-copy borrow; input
  must outlive the view), bounds-checked on every read (the C++ answer to C's manual checks).
  Base58 (`src/base58.cpp`) + LEB128 varint (`src/varint.cpp`) hand-rolled.
- **Crypto:** Ed25519 + SHA-256 via **libsodium 1.0.22** (`crypto_sign_*` / `crypto_hash_sha256`,
  `src/crypto.cpp`); the all-zero-seed → RFC-8032 TEST-1 pubkey KAT passes. **Ed448 / SHA-384 agility
  is deferred** (A-CPP-002) — libsodium has no Ed448; the §9.1 floor (Ed25519 + SHA-256) is fully
  native and is the only path the corpus exercises.
- **Identity / peer_id:** §1.5 canonical-form construction (Ed25519 → `hash_type=0x00`
  identity-multihash, digest *is* the raw 32-byte pubkey), per the §1.5 v7.65 table, **not** the
  stale §7.4 SHA-256 pseudocode (the 6th+ spec-first corroboration, PRE-RESOLVED P1). Lowercase
  `std::format("{:02x}")` hex everywhere (C++ is lowercase by default; pinned explicitly in
  `profile.toml [idiom].hex_lowercase` to dodge the A-CL-009 uppercase address-space-path trap). The
  seed-`0x11` peer_id `2KHoAk7A5JmhygZJAdBua8iRD1CnBoJRfUBHgZeXNRTeFg` is **byte-identical to the
  cohort**.
- **Error model:** the **`std::expected<T, E>` value channel** — every fallible operation returns an
  `expected` carrying either the result or a typed error; exceptions are reserved for
  programmer-error only, never the hot path. Protocol status is carried as a value, never smuggled
  across an error return. §5.2 request verification is a three-way verdict (ALLOW / AUTHN_FAIL→401 /
  AUTHZ_DENY→403) + the §5.5 unresolvable-grantee→401 carve-out.
- **Memory (the C++ seam vs C):** **RAII** — `std::unique_ptr`/`std::shared_ptr`, move semantics,
  deterministic destructors; no manual `new`/`delete`/`free`. Shared entity lifetime is
  `std::shared_ptr<const Entity>` — an atomic refcount, so the **A-C-009 race the C peer surfaced
  (a plain-`int` refcount freed a still-referenced entity under live §7b load) is structurally
  pre-resolved** (A-CPP-011). `-fvisibility=hidden` + an `ENTITY_CORE_API` export macro hide
  internals; libsodium symbols are localized so an embedder's own libsodium does not collide.
- **Concurrency (§7b):** one reader thread per connection demuxing `EXECUTE_RESPONSE` by
  `request_id`, one thread per inbound EXECUTE (§4.8 / §6.11), a `std::shared_mutex` guarding the
  content store, a per-connection write mutex, `TCP_NODELAY` on every connection socket. All 5 §7b
  concurrency checks PASS. (`libtsan` is absent from the image, so the ThreadSanitizer pass is not
  runnable in-container — A-CPP-013; the §7b live gate + the structural `shared_ptr`/`shared_mutex`
  design carry the data-race coverage.)

## Build / test / run-conformance (in-container, offline)

All build/test/conformance runs happen in the pinned `cpp-toolchain` Podman image (`fedora:43` base +
**gcc-c++ 15.2.1** + **clang 21.1.8** + **CMake 3.31.11** + **Ninja 1.13.1** + **libsodium 1.0.22**
static+devel + libasan/libubsan). libsodium is pre-installed in the image, so every run is fully
sealed-offline (`--network=none`). There is **no host g++/cmake/libsodium dependency**. The build is
**CMake + Ninja** (the profile slate's named build system — A-CPP-003; unlike the C peer's plain
`make`).

> keeping the headline `std::expected` idiom with **zero** new deps. `<span>`/`<format>` work under
> C++20; only `<expected>` forces the bump.

## Consume (the CMake package)

The stable public surface is the umbrella header **`include/entity_core/protocol.hpp`** plus the
`entity_core::` API; everything else is hidden (`-fvisibility=hidden` + `ENTITY_CORE_API`). The
install lays down the static + shared `libentity_core_protocol`, the headers, and a CMake package
config so consumers `find_package` it:

```cmake
# After `cmake --install build --prefix <prefix>` (or via a package manager below):
find_package(EntityCoreProtocol 0.1.0 REQUIRED)
target_link_libraries(myapp PRIVATE entity_core::protocol)
```

```cpp
#include <entity_core/protocol.hpp>

// Tier 1 — the codec/crypto island (the public API).
entity_core::crypto_init();
auto e   = entity_core::EcfValue::map();              // build an ECF value tree
auto bytes = entity_core::ecf_encode(e);              // -> std::expected<std::vector<std::byte>, Error>
auto back  = entity_core::ecf_decode(*bytes);         // structural decode (zero-copy span borrow)
auto pid   = entity_core::peer_id_from_pubkey(0x01, pub);  // §1.5 canonical peer_id (std::expected)
```

Three consumption forms ship — direct CMake (`FetchContent` / `add_subdirectory` / system install),
the **vcpkg port** (`packaging/vcpkg/`), and the **conan recipe** (`packaging/conan/conanfile.py`):

```sh
# vcpkg (decentralized port — overlay or registry):
vcpkg install entity-core-protocol --overlay-ports=packaging/vcpkg

# conan (decentralized recipe):
conan create packaging/conan --version 0.1.0
```

The **full peer + transport** (Tier 2 — `Peer`, `Transport`, the store + capability surface) is
driven via the **`host`** binary (`test/host.cpp` → `host --port N [--name NAME]
[--debug-open-grants] [--validate]`, emits `LISTENING …`). That peer-layer surface is **not** part
of the frozen public header at `0.1.0-pre` — exposing it as a stable Tier-2 ABI (via the
`opaque_pimpl` boundary the profile names) is a mechanical publish-prep pass, deferred until the
surface is frozen against a first external consumer (the honest `-pre` state; mirrors the C
`entity-peer-c` / OCaml `.mli` / Zig `root.zig` deferral). Internal units may churn without a semver
bump.

## Distribution (when published)

C++ has **no central package registry** (no crates.io/npm/Maven). "Publishing" is therefore three
parallel decentralized surfaces, all authored here and version-pinned at `0.1.0`:
1. an installable **CMake package** (`EntityCoreProtocolConfig.cmake`, generated by the install +
   `CPack`) consumed via `find_package` / `FetchContent` / `add_subdirectory`;
2. a **vcpkg port** (`packaging/vcpkg/{portfile.cmake,vcpkg.json}`);
3. a **conan recipe** (`packaging/conan/conanfile.py`).
`cmake --build build --target package` (CPack) produces the versioned source/binary package. **None
of these are pushed to any registry** — that is an operator action after review (lifecycle
§Publishing); there is no publish command run by the pipeline.
