## What it is

A native Java peer with a **zero-runtime-dependency core**: canonical-CBOR (ECF) codec, Ed25519
**and Ed448** identity + deterministic signatures, capability authorization with chain
attenuation + §5.7 delegation caveats, the §4.1 handshake / §6.5–§6.6 dispatch machinery, the
§9.5 53-type registry, the §9.5a CORE-TREE get/put/CAS/delete surface, and the v7.73/v7.74 peer
surface (§6.13 register, §PR-8 granter frame, §6.9a owner-cap, §7a conformance handlers). No FFI,
no BouncyCastle in the shipped artifact — `mvn` pulls nothing onto the runtime classpath.

- **Codec:** hand-rolled canonical CBOR (`...protocol.codec`) — shortest-float minimization
  (f16 ⊂ f32 ⊂ f64, narrowest bit-exact round-trip), length-then-lex map-key sort on encoded key
  bytes, recursive major-type-6 tag rejection on decode, CBOR head-form integer carrier via
  `BigInteger` for the full 0..2⁶⁴−1 range (Java's `long` has no native unsigned, so `BigInteger`
  is the value carrier — sidestepping the C# `ulong`/OCaml int63 trap by a different mechanism).
  Base58 + the multicodec LEB128 varint are hand-rolled (neither is in the JDK).
- **Crypto:** Ed25519 + Ed448 sign/verify via the **JDK SunEC provider** (zero-dep, native);
  SHA-256/384 via `MessageDigest`. The JDK's vendor-curated stdlib closes the crypto-agility
  *signature* bar natively. **The one gap (A-JAVA-007):** SunEC exposes no seed→public-key API
  and the JDK ships **no SHAKE256** XOF, so the §1.5 identity-multihash *raw public key* needs a
  hand-rolled, KAT-verified FIPS-202 SHAKE256 + RFC-8032 scalar-mult derivation (~250 lines, pure
  JDK, no FFI). BouncyCastle (`bcprov-jdk18on` 1.80) is `provided`-scope test-only cross-check —
  the core build stays BouncyCastle-free. See
  [`status/SPEC-AMBIGUITY-LOG.md`](status/SPEC-AMBIGUITY-LOG.md) (A-JAVA-002, A-JAVA-007).
- **Identity / peer_id:** §1.5 canonical-form construction (Ed25519 → `hash_type=0x00`
  identity-multihash raw pubkey; Ed448 → `hash_type=0x01` SHA-256-of-pubkey), following the §1.5
  v7.65 table, **not** the stale §7.4 SHA-256 pseudocode (A-JAVA-004). The seed-`0x11` peer_id
  `2KHoAk7A5JmhygZJAdBua8iRD1CnBoJRfUBHgZeXNRTeFg` is **byte-identical to the Common Lisp peer**.
  Lowercase `%02x` hex everywhere (dodges the A-CL-009 uppercase address-space-path trap).
- **Error model:** Java **checked exceptions** — a typed exception lattice for codec/local
  faults; protocol status is carried as a value record, never across an exception (the cohort's
  status-as-value invariant). §5.2 request verification returns a three-way verdict
  (ALLOW / AUTHN_FAIL→401 / AUTHZ_DENY→403) + the §5.5 unresolvable-grantee→401 carve-out
  (A-JAVA-009).
- **Concurrency (the JDK-21 seam):** **virtual threads** (Project Loom, JEP 444 GA in 21) — one
  reader virtual-thread per connection demuxing `EXECUTE_RESPONSE` by `request_id`, one virtual
  thread per inbound EXECUTE (§4.8 / §6.11), a `ConcurrentHashMap<requestId, …>` rendezvous
  table. Thread-per-connection-*and*-per-request is cheap on Loom — the model other peers
  justified *against* OS-thread cost is the *recommended* carrier here (A-JAVA-003). All 5 §7b
  concurrency checks (demux, no-head-of-line, sustained load, connection churn) PASS.

## Build / test / run-conformance (in-container, offline)

All build/test/conformance runs happen in the pinned `java-toolchain` Podman image (`fedora:43`
base + **Temurin JDK 21 LTS** SHA-256-pinned + Apache **Maven 3.9.9** SHA-512-pinned). Maven
deps (JUnit 5, BouncyCastle, plugins) are pre-fetched into the image `~/.m2` at container-build
time, so every run is fully sealed-offline (`mvn -o`, `--network=none`). There is **no host Java
or Maven dependency**.

## Use

Public surface is the `org.entitycore.protocol` package tree. Two tiers:

```java
import org.entitycore.protocol.codec.*;   // Tier 1 — codec island
import org.entitycore.protocol.crypto.*;  // Tier 1 — identity + signatures
import org.entitycore.protocol.peer.*;    // Tier 2 — full peer + transport

// Tier 1 — encode/verify ECF (no peer)
Entity e = Entity.make("system/peer", dataMap);     // dataMap: EcfValue.Map
byte[] bytes = Cbor.encode(e.toCbor());             // canonical ECF bytes
Identity id = Identity.ofSeed(seed);                // Ed25519 identity + §1.5 peer_id

// Tier 2 — full peer
Peer peer = Peer.create(seed, /*openGrants=*/ false);
new Transport(peer).acceptLoop(7777);              // virtual-thread per connection
```

Internal units (varint, base58, wire framing, the type-registry render table) are implementation
detail and may churn without a semver bump. An explicit module-info / public-surface freeze is a
mechanical publish-prep pass, deferred until the surface is frozen against a first external
consumer — the honest `0.1.0-pre` state (mirrors the OCaml `.mli` / Zig `root.zig` deferral).

## Distribution (when published)

Java's registry is **Maven Central** (Sonatype Central Portal). "Publishing" is
`mvn -o -B deploy` to a staging repository against a **verified `org.entitycore` reverse-DNS
namespace** — the namespace verification is a one-time operator step that gates the first deploy
(A-JAVA-005), and Maven Central publishing is therefore **deferred**, not done here. The artifact
is publish-*ready*: well-formed `pom.xml` with `org.entitycore:entity-core-protocol:0.1.0-pre`
coordinates, license metadata, and a zero-dependency runtime classpath. See
[`status/PHASE-S5.md`](status/PHASE-S5.md) §Operator-handoff and
[`status/ARCHITECTURE-REVIEW.md`](status/ARCHITECTURE-REVIEW.md) §Publishing-options for the
in-repo-vs-standalone-repo decision surface (operator-decides; not acted on by the pipeline).
