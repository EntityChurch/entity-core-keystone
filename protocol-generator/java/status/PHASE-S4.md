## The grind: 86 → 0 FAIL (iteration count: 3 oracle runs)

**Run 1 (baseline, S3 peer as-is):** 573 · 200P / 195W / **86 FAIL** / 92S. Two fail
clusters — all derived spec-first from V7 + the cohort, never by doctoring the oracle,
hand-editing vectors, or disguising a FAIL as a SKIP:

1. **type_system: 85 FAIL** (A-JAVA-008) — the full §9.5 floor wasn't published (S3 seeded
   only ~14 stub types); fetches like `system/type/core/entity` 404'd. **Fix:** generated
   `CoreTypeDefs` (the 53-type override table) from the shared cross-impl type model, wired
   `CoreTypes.publish` to render all 53 from it. Each content_hash computed by our own
   S2-green codec → 53/53 byte-identical first render.
2. **concurrency `t1_2_concurrent_reentry`: 1 FAIL** — the §7a dispatch-outbound handler
   re-wrapped the relayed `value` (`{value: value}`), double-nesting so the validator's
   `echoed.value` decoded as a MAP not a string. **Fix:** make it a generic relay — forward
   the `{value: X}` params map through unwrapped, return the downstream result entity
   verbatim (the §7b concurrency-gate matrix ruling #2; A-JAVA-011).

**Run 2:** 573 · 286P / 195W / **0 FAIL** / 92S — but `Result: FAIL (un-allowlisted skips)`
from **3 SKIPs**: concurrency `t1_1`/`t1_3`/`t2_1`, whose `tree.put` to
`system/validate/concurrency/*` was returning a silent **500**. Root cause (A-JAVA-010): the
Java `Entity` modeled `data` as `EcfValue.Map`, but those checks stage **`primitive/string`**
entities — **scalar** data. `Entity.ofCbor` rejected non-map data → 500. This was a latent
peer data-model bug (S2 corpus + S3 loopback only ever exercised map-data entities). **Fix:**
generalize `Entity` to hold `data` as an arbitrary `EcfValue` (§1.1), keep a null-safe map
view for the protocol field-readers.

**Run 3:** 573 · **289P / 195W / 0 FAIL / 89S → PASS.** All 5 concurrency checks PASS.
